 Document 1: Harness Engineering 概念导论

  This is a comprehensive academic/introductory document about Harness Engineering. Key insights:

  Core Concept: What is Harness?

  - The "harness" metaphor comes from test harness: providing input, environment, observation, and assertions for the model
  - "Agent的表现不是模型单独的属性，而是模型、Harness与环境共同作用的结果"
  - Harness sits between the model and the execution environment

  Key Architectural Insight: 6-Layer Model

  - L0: Model (能力基础，不是Harness本体)
  - L1: 指导与上下文 (Prompt, docs, Skills, RAG, Progress)
  - L2: 行动与环境 (Tools, MCP, Workspace, Sandbox)
  - L3: 任务运行与服务 (Loop, 编排, 状态, 记忆, 恢复, 消息和Streaming)
  - L4: 质量与交付 (Event logs, Trace, Tests, Eval, CI, Review)
  - L5: 治理与组织 (身份, 审批, 成本, 审计, 职责)

  The "四问法" (Four Questions for Boundary Judgment)

  1. T1: 运行时是否存在"推理/决策—行动—观察"的闭环？
  2. T2: 模型是否能通过接口感知并改变外部环境？
  3. T3: 系统是否主动、按任务内容管理上下文和状态？
  4. T4: 是否至少有一种不依赖"模型自觉服从"的控制或验证？

  Key Problems for Agent Tasks

  1. 输出变成行动
  2. 一次调用变成持续循环
  3. 外部环境成为任务状态的一部分
  4. 上下文窗口不再等于任务记忆
  5. 最终回复不再等于交付结果
  6. 调用接口变成长期服务

  Anthropic Long Task Case - Key Insights

  - Even with strong model + loop + compaction, long tasks fail without external state management
  - Solutions: feature_list.json, claude-progress.txt, Git history, init.sh, end-to-end tests
  - "完成"由外部验收结果决定，不由模型一句话决定

  Three Engineering Observations

  1. SWE-agent: Fixed model, changing interface → task results change (18% vs 11%)
  2. Anthropic: Strong model + loop + compaction → still can't maintain long tasks naturally
  3. OpenAI: Agent-first repository transforms "things the model can't see" into system objects

  Four Engineering Conclusions

  1. 任务从可验证的契约开始
  2. 上下文窗口不是任务状态
  3. 能用确定性机制表达的约束，不应只写成自然语言
  4. 重要失败要进入可重复的回归闭环

  Document 2: Hermes Agent 自进化长期记忆

  Core Insight: 记忆是治理问题，不是存储问题

  - RAG只覆盖召回；写入、修正、遗忘决定"这条记录未来该不该影响行为"
  - 数据库负责"存进去还能读出来"；记忆系统还要负责"这条记录在未来该不该影响行为"

  Four-Layer Memory Model

  1. Working Memory: 当前任务必要信息 → task_brief.json
  2. Raw Episodic: 原始事件、工具输出、对话 → raw_events.jsonl
  3. Consolidated Episodic: 任务摘要、决策、失败原因 → consolidated_sessions.jsonl
  4. Semantic / Policy: 稳定规则、偏好、长期事实 → semantic_rules.jsonl

  Hermes Agent Structure Reference

  - MEMORY.md / USER.md: Semantic/Policy稳定规则
  - SQLite + session_search: Raw Episodic完整事件流
  - 会话结束抽取/后台review: Raw→Consolidated→Semantic整理动作
  - Memory Provider: 语义检索与外部记忆扩展

  Pipeline: Extract → Classify → Merge → Prune → Index

  1. Extract: 从Task steps中提取关键信息
  2. Classify: fact / procedure / reflection
  3. Merge: 与已有记忆比对（新/冲突/补充/重复）
  4. Prune: 超过30天未访问→降级，被覆盖→obsolete，与事实不符→失效
  5. Index: embedding向量化+关键词索引

  Three Promotion Channels (晋升通道)

  1. cross-session repetition (跨会话重复)
  2. tool failure evidence (工具失败证据)
  3. explicit user instruction (用户显式长期指令)

  Heartbeat Mechanism

  - 后台定时任务，扫描新raw事件→触发抽取→合并→晋升→checkpoint
  - Adaptive interval: new>=10 → 60s, new>=1 → 300s, none → 900s
  - 分工原则: 模型负责理解自然语言的判断，代码负责可核对的结构化判断

  Security: Memory Pollution

  - Background heartbeat can silently pollute memory through E→M→B path
  - 后台写入必须有独立治理：来源标记、证据链接、置信度、可回滚

  Retrieval Pipeline

  1. 关键词匹配 (快速)
  2. 向量相似度 (语义)
  3. 相关性排序 (融合): score = 0.6×向量相似度 + 0.2×关键词匹配 + 0.1×新鲜度 + 0.1×历史命中率
  4. 注入上下文

  Key Engineering Boundary

  | 角色 | 负责 | 不负责 |
  | Session | 当前会话历史、窗口裁剪 | 跨任务长期记忆治理 |
  | 文件记忆库 | raw/consolidated/semantic/task_brief | 工程级权限、索引、并发 |
  | Workspace | 工程化持久化、scope隔离、ContextPackage | 四层记忆理论本身 |
  | 向量库扩展 | 语义近义召回加速 | 主事实存储、证据链、权限 |
  | 心跳整理器 | 定时触发抽取、合并、晋升、checkpoint | 记忆理论和业务规则 |
  | 模型 | 抽取、同义判断、摘要、表达 | 独立完成记忆治理 |
