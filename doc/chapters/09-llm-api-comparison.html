LLM API 厂商对比 参考文档

深入对比 OpenAI、Anthropic、DeepSeek、阿里 DashScope 四家 LLM 厂商的 API 设计差异
为后续 Provider 接口抽象提供设计参考

📋 文档目的
本文档分析各厂商 API 的请求/响应格式差异、独有特性、流式协议差异， 为 Phase 5-6 的 Provider 接口抽象层提供设计依据。 当前项目仅实现了 OpenAI-compatible 协议（覆盖 DeepSeek、阿里百炼等国产模型）。

1. OpenAI API 完整列表

OpenAI 提供的 API 端点远不止 /chat/completions。以下按与本项目的相关性排列：

1.1 核心 API

端点方法用途本项目
/v1/chat/completions POST 对话补全 — 支持文本、工具调用、多模态 ✅ 已实现
/v1/embeddings POST 文本向量化 — 将文本转为高维向量 🔮 Phase 6 (RAG)
/v1/models GET 列出可用模型及其元信息 🔮 可选

1.2 辅助 API

端点方法用途本项目
/v1/completions POST 文本补全（Legacy，GPT-3 时代） ❌ 不需要
/v1/audio/transcriptions POST 语音转文字 (Whisper) ❌ 不需要
/v1/audio/speech POST 文字转语音 (TTS) ❌ 不需要
/v1/images/generations POST 图片生成 (DALL·E) ❌ 不需要
/v1/files GET/POST/DELETE 文件上传/管理（用于 Fine-tuning） ❌ 不需要
/v1/fine_tuning/jobs POST 微调任务管理 ❌ 不需要
/v1/moderations POST 内容审核 ❌ 不需要
🔑 关键结论
与本项目相关的 OpenAI API 只有三个：Chat Completions（已实现）、 Embeddings（Phase 6 RAG）、Models（可选）。 其余 API 要么已废弃，要么与本项目领域无关。

2. Chat Completions API 参数详解

OpenAI Chat Completions API 是当前行业事实标准，绝大多数国产模型兼容此协议。 以下列出完整参数（加粗参数为当前已实现）：

2.1 请求参数

参数类型必填默认值说明
model string✅— 模型标识符，如 gpt-4、deepseek-v4-flash
messages array✅— 对话历史，按角色排列：system → user → assistant → tool → …
stream bool❌false 是否启用 SSE 流式返回
temperature float❌1.0 采样温度 (0-2)，越高越随机
max_tokens int❌— 最大输出 token（已废弃，建议用 max_completion_tokens）
tools array❌— Function Calling 工具定义列表
tool_choice string/obj❌"auto" 工具选择策略：auto / none / required / 指定工具
max_completion_tokens int❌— 新版参数，替代 max_tokens，仅限制输出 token
top_p float❌1.0 Nucleus sampling — 从累积概率 top_p 的 token 集合中采样
stop string/array❌— 停止词 — 遇到即停止生成
frequency_penalty float❌0 频率惩罚 (-2.0 ~ 2.0)，降低重复
presence_penalty float❌0 存在惩罚 (-2.0 ~ 2.0)，增加话题多样性
n int❌1 生成几个候选回复
seed int❌— 确定性输出种子（相同 seed + 相同参数 = 相同输出）
response_format object❌— 结构化输出：{"type": "json_object"} 或 {"type": "json_schema", "json_schema": {...}}
logprobs bool❌false 是否返回每个 token 的 log 概率
top_logprobs int❌— 返回 top N 个 token 的概率（需 logprobs=true）
parallel_tool_calls bool❌true 是否允许 LLM 并行调用多个工具
stream_options object❌— {"include_usage": true} — 在流式最后一个 chunk 中返回 usage
user string❌— 终端用户标识符（用于 OpenAI 监控和滥用检测）
⚠️ max_tokens vs max_completion_tokens
OpenAI 已将 max_tokens 标记为废弃，推荐使用 max_completion_tokens。 但部分国产模型（如 DeepSeek）仍使用 max_tokens。
建议：Phase 5 Provider 抽象时同时支持两个参数，按厂商适配。

2.2 SSE 流式返回格式

// 每个 SSE chunk 的 JSON 结构
{
  "id": "chatcmpl-xxx",
  "object": "chat.completion.chunk",
  "created": 1234567890,
  "model": "gpt-4",
  "choices": [{
    "index": 0,
    "delta": {
      "role": "assistant",           // 仅第一个 chunk 有此字段
      "content": "Hello",            // 文本增量（1~N 个 token）
      "tool_calls": [{               // 工具调用增量（分多个 chunk 到达）
        "index": 0,
        "id": "call_xxx",            // 仅第一个 tool delta 有
        "type": "function",
        "function": {
          "name": "run_shell",       // 仅第一个 delta 有
          "arguments": "{\"com"      // 增量 JSON 片段
        }
      }]
    },
    "finish_reason": null            // 最后一个 chunk 有值: "stop"/"tool_calls"/"length"
  }],
  "usage": {                         // 仅 stream_options.include_usage=true 时返回
    "prompt_tokens": 10,
    "completion_tokens": 20,
    "total_tokens": 30
  }
}

2.3 ToolCall 增量到达顺序

ToolCall 的增量是乱序到达的，这是 SSE 协议的核心挑战：

// Chunk 1: 索引和 ID 先到
delta: { tool_calls: [{ index: 0, id: "call_abc", type: "function" }] }

// Chunk 2: 函数名到达
delta: { tool_calls: [{ index: 0, function: { name: "run_shell", arguments: "" } }] }

// Chunk 3-N: 参数片段逐步到达（可能跨多个 chunk）
delta: { tool_calls: [{ index: 0, function: { arguments: "{\"com" } }] }
delta: { tool_calls: [{ index: 0, function: { arguments: "mand" } }] }
delta: { tool_calls: [{ index: 0, function: { arguments: "\": \"ls" } }] }
delta: { tool_calls: [{ index: 0, function: { arguments: " -la\"}" } }] }

// 最后一帧: finish_reason 出现
delta: {}, finish_reason: "tool_calls"

当前实现使用 map[int]*ToolCall 按 index 累积，组装完成后转为有序切片。

3. Anthropic (Claude) Messages API

Anthropic 的 API 设计与 OpenAI 完全不兼容，有自己独立的协议体系。 核心差异在于：system prompt 的传递方式、工具 schema 字段名、流式协议格式。

3.1 核心端点

端点方法用途
/v1/messagesPOST对话（等价于 OpenAI 的 /chat/completions）
/v1/messages/count_tokensPOST计算 token 数（不实际调用 LLM）

3.2 请求格式

// Anthropic Messages API 请求体
{
  "model": "claude-sonnet-4-6",
  "max_tokens": 1024,              // ⚠️ 必填！跟 OpenAI 最大区别
  "system": "You are a helpful assistant",  // ⚠️ 顶层字段，不在 messages 数组里
  "messages": [
    {"role": "user", "content": "Hello"},
    {"role": "assistant", "content": "Hi!"},
    {"role": "user", "content": [
      {"type": "text", "text": "What's in this image?"},
      {"type": "image", "source": {
        "type": "base64",
        "media_type": "image/png",
        "data": "iVBORw0KGgo..."
      }}
    ]}
  ],
  "tools": [{
    "name": "get_weather",
    "description": "Get the current weather",
    "input_schema": {              // ⚠️ 字段名是 input_schema，不是 parameters
      "type": "object",
      "properties": {
        "location": {"type": "string", "description": "City name"}
      },
      "required": ["location"]
    }
  }],
  "tool_choice": {"type": "auto"}, // ⚠️ 对象格式，不是字符串
  "stop_sequences": ["\n\nHuman:"], // ⚠️ 数组，不是 stop 字符串
  "temperature": 0.7,
  "top_p": 0.9,
  "top_k": 40,                     // ⚠️ Anthropic 独有参数
  "stream": true
}

3.3 流式返回格式

Anthropic 的 SSE 格式与 OpenAI 完全不同——使用 event: 行标识事件类型：

// === Anthropic SSE 事件流 ===

event: message_start
data: {"type": "message_start", "message": {"id": "msg_xxx", "usage": {"input_tokens": 10}}}

event: content_block_start
data: {"type": "content_block_start", "index": 0, "content_block": {"type": "text", "text": ""}}

event: content_block_delta
data: {"type": "content_block_delta", "index": 0, "delta": {"type": "text_delta", "text": "Hello"}}

// 工具调用以独立 content_block 出现
event: content_block_start
data: {"type": "content_block_start", "index": 1, "content_block": {
  "type": "tool_use", "id": "toolu_xxx", "name": "run_shell", "input": {}
}}

event: content_block_delta
data: {"type": "content_block_delta", "index": 1, "delta": {
  "type": "input_json_delta", "partial_json": "{\"com"
}}

event: content_block_stop
data: {"type": "content_block_stop", "index": 1}

event: message_delta
data: {"type": "message_delta", "delta": {"stop_reason": "tool_use"}, "usage": {"output_tokens": 50}}

event: message_stop
data: {"type": "message_stop"}
⚠️ Anthropic 流式协议的关键差异
  • 双层结构：event: type + data: {...}，而非 OpenAI 的单层 data: {...}
  • Tool use 是独立 content_block：不是 delta.tool_calls 数组，而是独立的事件类型
  • 参数是 partial_json：不是 function.arguments 增量，而是 input_json_delta
  • Usage 分两段：input_tokens 在 message_start 中，output_tokens 在 message_delta 中

3.4 与 OpenAI 的关键差异总结

维度OpenAIAnthropic
System Prompt 在 messages 数组里 (role: "system") 顶层 system 字段
工具参数 Schema function.parameters input_schema
max_tokens 可选 必填
多模态 Content [{type: "text"}, {type: "image_url"}] [{type: "text"}, {type: "image", source: {...}}]
停止词 stop: string | array stop_sequences: array
流式格式 data: {"choices": [...]} 单层 event: type + data: {...} 双层
Tool Use 流式 delta.tool_calls[].function.arguments 增量 delta.partial_json 增量
采样参数 temperature + top_p temperature + top_p + top_k
Token 计数 usage 在最后一个 chunk input 在 message_start，output 在 message_delta
Tool Choice "auto" | "none" | "required" {"type": "auto"} 对象格式

4. DeepSeek API

✅ 完全兼容 OpenAI 协议
DeepSeek 的 API 100% 兼容 OpenAI Chat Completions 协议，无需任何代码修改即可使用当前 internal/llm/client.go。

4.1 核心端点

端点方法说明
/v1/chat/completionsPOST标准对话补全，完全兼容 OpenAI
/v1/modelsGET模型列表

4.2 独有特性：reasoning_content

DeepSeek R1 等推理模型会返回思维链内容（thinking process）， 在 delta 中与 content 并列返回：

// DeepSeek R1 的 SSE chunk 扩展
{
  "choices": [{
    "index": 0,
    "delta": {
      "reasoning_content": "Let me think about this step by step...\n\nFirst, we need to...",  // 🆕 思考过程
      "content": ""                                                                             // 正式回答
    }
  }]
}

// 思考完成后，才开始输出正式回答
{
  "choices": [{
    "index": 0,
    "delta": {
      "reasoning_content": "",                  // 思考结束
      "content": "The answer is 42"             // 正式回答开始
    }
  }]
}
🔮 扩展计划
reasoning_content 可用于实现"展示 Agent 思考过程"的白盒可视化特性。 计划在 Phase 5-6 的 Provider 抽象中作为可选字段支持。 当前代码中 Delta 结构体未包含此字段，需要向后兼容地扩展。

4.3 其他注意事项

  • 使用 max_tokens 而非 max_completion_tokens
  • 不支持 parallel_tool_calls 参数
  • 部分模型（如 deepseek-chat）支持 frequency_penalty 和 presence_penalty
  • tool_choice 支持 "auto" 和 "none"，不支持 "required"

5. 阿里 DashScope (百炼)

阿里 DashScope 同时提供原生协议和OpenAI 兼容模式。 兼容模式下基本可用，但有一些独有参数和响应格式差异。

5.1 核心端点

端点方法说明
/compatible-mode/v1/chat/completionsPOSTOpenAI 兼容模式（推荐）
/v1/services/aigc/text-generation/generationPOST原生文本生成
/v1/services/embeddings/text-embedding/text-embeddingPOST文本向量化

5.2 独有参数

参数类型说明
enable_search bool 联网搜索增强 — 模型可实时搜索互联网
repetition_penalty float 重复惩罚系数（非 OpenAI 标准参数）
result_format string 返回格式："message" 或 "text"
incremental_output bool 是否增量输出（流式专用）

5.3 响应格式差异

阿里原生协议的 SSE 响应结构与 OpenAI 有细微差异：

// 阿里 DashScope 原生 SSE 格式
{
  "output": {                          // ⚠️ 外层包了 output
    "choices": [{
      "message": {                     // ⚠️ 用 message 而非 delta
        "content": "Hello",
        "role": "assistant"
      },
      "finish_reason": "null"
    }]
  },
  "usage": {
    "output_tokens": 10,               // ⚠️ 字段名是 output_tokens 而非 completion_tokens
    "input_tokens": 20
  },
  "request_id": "xxx"
}
⚠️ 兼容模式 vs 原生模式
建议使用阿里 DashScope 的OpenAI 兼容模式（/compatible-mode/v1/chat/completions）， 该模式下请求/响应格式与 OpenAI 基本一致。
原生模式仅当需要 enable_search 等独有特性时才使用。

7. Provider 接口抽象设计建议

为支持多厂商 LLM，需要从当前硬编码的 OpenAI-compatible 客户端中抽取一层 Provider 接口。 以下为建议的架构设计：

7.1 目标目录结构

internal/llm/
├── client.go          # 当前: OpenAI-compatible 通用客户端 (保留)
├── provider.go        # 新增: Provider 接口 + 工厂函数
├── openai_provider.go  # 新增: OpenAI provider (基线实现)
├── anthropic_provider.go  # 新增: Anthropic provider (Phase 6)
└── deepseek_provider.go   # 新增: DeepSeek provider (reasoning_content 扩展)

7.2 Provider 接口设计

// Provider 抽象不同 LLM 厂商的 API 差异，将通用请求转换为厂商特定格式。
// 接口设计遵循"统一输入、统一输出"原则：
//   - ChatRequest / StreamDelta 是项目内部的统一类型
//   - 每个 Provider 负责将其与厂商 API 格式互转
type Provider interface {
    // Name 返回厂商名称，如 "openai"、"anthropic"、"deepseek"
    Name() string

    // Endpoint 返回 API 基础地址
    Endpoint() string

    // BuildRequest 将统一请求转为厂商特定的 HTTP body
    BuildRequest(req *ChatRequest) ([]byte, error)

    // ParseStreamChunk 解析一行 SSE 数据，返回统一格式的 StreamDelta
    // 不同厂商的 SSE 格式差异在此方法中处理
    ParseStreamChunk(line []byte) (*StreamDelta, error)

    // BuildHeaders 返回厂商特定的 HTTP 请求头
    BuildHeaders() map[string]string
}

// StreamDelta 是项目内部的统一流式增量格式
type StreamDelta struct {
    Content          string     // 文本增量
    ToolCalls        []ToolCallDelta  // 工具调用增量
    FinishReason     string     // 停止原因
    ReasoningContent string     // 思维链内容 (DeepSeek R1 等)
    Usage            *Usage     // token 使用量
}

// ToolCallDelta 是工具调用的增量格式
type ToolCallDelta struct {
    Index     int
    ID        string
    Name      string
    Arguments string  // 增量 JSON 片段
}

7.3 工厂函数

// NewProvider 根据厂商名称创建对应的 Provider 实例
func NewProvider(name string, cfg *ProviderConfig) (Provider, error) {
    switch name {
    case "openai":
        return NewOpenAIProvider(cfg)
    case "anthropic":
        return NewAnthropicProvider(cfg)
    case "deepseek":
        return NewDeepSeekProvider(cfg)
    default:
        // 默认回退到 OpenAI-compatible
        return NewOpenAIProvider(cfg)
    }
}

7.4 设计原则

  1. 统一输入输出：Engine 层只与 ChatRequest / StreamDelta 交互，不感知厂商差异
  2. 每厂商一个 Provider：每个 Provider 文件负责一个厂商的协议转换
  3. 默认回退：未知厂商自动走 OpenAI-compatible 路径
  4. 向后兼容：现有的 Client 结构体在迁移期间保持不变
  5. 可扩展字段：StreamDelta 预留 ReasoningContent 等扩展字段

8. 实施计划

阶段内容优先级
Phase 2-3 保持现有 OpenAI-compatible 客户端不动
理由：DeepSeek、阿里百炼、智谱、月之暗面等国产模型全兼容此格式 		无需改动
Phase 3-4 扩展 Delta 结构体，增加 ReasoningContent 字段
理由：DeepSeek R1 / Qwen3 等推理模型的思维链展示 		低
Phase 5 抽取 Provider 接口，创建 openai_provider.go
理由：为 Runtime Tool 注册等功能提供更灵活的 LLM 配置 		中
Phase 6 实现 AnthropicProvider + DeepSeekProvider
理由：作为高级特性，支持多厂商 LLM 切换 		高
Phase 6 实现 Embeddings API（RAG 向量检索）
理由：向量检索是 RAG 的基础 		高
💡 为什么不现在就做 Provider 抽象？

当前阶段（Phase 2-3）的核心任务是前端可视化和预设 Cases， 国产模型几乎全部兼容 OpenAI 协议，增加 Anthropic 支持属于过早抽象。

等到 Phase 5-6 需要 Runtime 注册、RAG 等高级特性时，再引入 Provider 接口层， 届时对 LLM 领域的理解更深，接口设计会更成熟。
