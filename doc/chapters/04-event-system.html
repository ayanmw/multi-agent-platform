4. 事件系统

AgentEvent 协议定义、通信机制、状态机流程。

4.1 设计目标

事件系统是整个平台的核心通信中枢，负责：

  • 将后端 Agent 执行状态实时推送到前端
  • 支持多 Agent 并行时的状态隔离与合并
  • 提供细粒度的事件类型，前端可按需展示或隐藏细节

4.2 Event 结构体

所有 WebSocket 消息统一使用 AgentEvent 结构：

{
  "event_id":   "a1b2c3d4e5f6",      // 唯一 ID（UUID fragment）
  "task_id":    "task_20260101000000", // 关联任务 ID
  "agent_id":   "agent_research_001",  // 事件来源 Agent
  "step_index": 0,                    // 当前第几轮 Loop
  "type":       "llm_delta",          // 事件类型
  "timestamp":  1717200000000,        // Unix ms
  "data": {                           // 事件载荷（类型相关）
    "content": "some text delta..."
  }
}

4.4 典型事件序列

一个完整的 ReAct Loop 产生的典型事件流：

1. task_started        → 任务开始
2.   step_started      → 第 0 轮开始 (type: think)
3.     llm_thinking    → LLM 开始生成
4.     llm_delta       → token 增量 × N
5.     llm_message_complete
6.   step_complete      → 第 0 轮结束（纯文本回复）
7.   step_started      → 第 1 轮开始 (type: tool_call)
8.     tool_call_started → 调用 write_file
9.     tool_call_input   → 参数确认
10.    tool_call_output  → 执行结果
11.    tool_call_complete → 完成（42ms）
12.  step_complete       → 第 1 轮结束
13.  observation        → Agent 观察：文件已写入
14. step_started        → 第 2 轮开始 (type: think)
...（循环直到最终答案）
N.  task_completed     → 任务完成，附带最终结果

4.6 事件去重与排序

注意事项
  • 每个事件有唯一 event_id，前端可用于去重
  • timestamp 保证事件有序（但 WebSocket 本身保证顺序）
  • 多 Agent 并行时，事件交错到达，前端按 agent_id 分桶
  • llm_delta 是批量事件，前端需追加而非覆盖

4.7 前端事件处理

前端收到事件后，按 type 分发到对应处理函数：

事件类型前端操作
task_started创建新 Task 节点，状态设为 running
step_started创建新 Step 节点，追加到对应 Agent
llm_delta追加 text 到最后 Step 的 thinking 字段
tool_call_*更新 Step 的 toolCall 对象
observation创建 observation 类型 Step
task_completed标记 Task 完成，显示 finalResult
