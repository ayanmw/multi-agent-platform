7.4 会话与任务历史管理（Phase 5）

目标：解决"完成任务后必须刷新页面才能回到主界面"的问题，并支持一个 Session 下多个 Task 的金字塔结构。

7.4.1 Session 概念

Session 是前端的一次对话上下文，在后端持久化。每个 Session 关联一个根 Task；根 Task 可以再派生子 Task。

  • 空 Session：刚创建，无根 Task，展示预设 Cases + 输入框。
  • 运行中 Session：根 Task 或任意子 Task 正在执行，实时显示 AgentTree。
  • 已完成/失败 Session：所有 Task 结束，可查看结果、Retry 或 Continue。

7.4.2 用户操作流程

  1. 点击左侧会话列表的 + New Session，创建空 Session，自动回到主界面。
  2. 输入任务或点击 Case Card，启动后端根 Task；Session 状态变为 running。
  3. 多 Agent 执行时，根 Task 下可能产生多个子 Task；前端通过 task_id 接收并渲染多棵树。
  4. 任务完成后，点击会话列表中的其他 Session，可以切换回看历史执行过程。
  5. 在历史失败 Session 上点击 Continue，用原始输入 + 新 max_steps 重新启动（创建新根 Task）。
  6. 页面刷新后，前端通过 GET /api/sessions 拉取历史 Session 列表。

7.4.3 前端状态调整

当前 useTaskStore 从单任务改为任务缓存 + 活跃任务 ID：

const taskCache = ref<Record<string, TaskState>>({})  // taskId -> TaskState
const activeTaskId = ref<string | null>(null)

新增 useSessionStore 管理 Session 列表、活跃 Session、新建/删除/切换：

const sessions = ref<Session[]>([])
const activeSessionId = ref<string | null>(null)

7.4.4 与现有功能的关系

  • lastUserInput（Phase 4+）：提升到 Session 级别，用于 Continue / Retry。
  • Continue with max steps ×2（Phase 4+）：在 Session 上下文内重新启动 Task。
  • Task 历史侧边栏（Phase 3 遗留）：升级为 Session 列表。
  • 后端 API：新增 /api/sessions / /api/sessions/:id，/api/tasks 增加 session_id 参数。
实现说明
Session 列表目前存储在后端 sessions 表，前端在 localStorage 做本地缓存；刷新页面后通过 API 重新拉取并合并。
