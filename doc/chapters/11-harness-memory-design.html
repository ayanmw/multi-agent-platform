11.1 Harness Engineering 概念引入

Harness Engineering 源自传统软件工程中的 test harness 隐喻：它不是被测对象本身，而是为被测对象提供输入、运行环境、观测和断言的一套外部装置。Agent harness 沿用了这个隐喻，只是对象从"被测程序"变成了"需要持续观察、行动和修正的模型"。

一句话定义
Harness 是包在模型周围的一层运行与控制系统，用行动循环、环境工具、上下文管理和独立控制，把模型变成能在外部环境中持续完成任务的 Agent。

为什么需要 Harness？

从单次模型调用到 Agent 任务，系统责任发生了质变：

阶段模型输出代表什么系统新增责任
单次请求一段文本或结构化结果输入、输出格式、超时、基本审计
Function Calling一个外部行动请求工具选择、参数校验、执行、错误回传
Agent Loop / ReAct下一步行动决策循环、观察、重试、停止条件、预算
Agent 任务对真实目标的持续推进工作空间、状态、记忆、权限、验证、恢复、服务化

三个来自工业界的观察，共同说明一件事：Agent 的任务表现是模型、接口、上下文/状态、控制机制与外部环境共同产生的系统属性，而非模型单独决定。

三条工程结论
  1. 任务从可验证的契约开始，而不是只从一段 Prompt 开始。目标、范围、证据、预算、权限和升级条件应当成为外部任务对象。
  2. 上下文窗口不是任务状态。计划、进度、决策、Workspace 和证据需要独立持久化，不能只保存在聊天历史里。
  3. 能用确定性机制表达的约束，不应只写成自然语言。Schema、Policy、Tests、权限和状态断言应尽量在模型之外执行。
  4. 重要失败要进入可重复的回归闭环。保存 Trace 和环境结果，定位失败归属，修改最接近根因的 Harness 层。

11.2 四问法评估当前项目

使用 Sanderson Oliveira de Macedo 的"四问法"（What makes a harness a harness, arXiv 2606.10106），对当前项目（Phase 1 完成态）进行边界评估：

问题通过条件当前状态缺口
T1: 运行时闭环？ 上一步的环境观察能改变下一步行动 ✅ ReAct Loop 完整实现
think → tool_call → observe → loop		 —
T2: 感知并改变环境？ 工具不只返回知识，还能写文件、运行命令 🔶 3 个内置工具
run_shell / write_file / read_file		 无沙箱隔离；无浏览器/网络工具；工具集固定不可扩展
T3: 任务感知上下文？ 进入模型窗口的信息由当前任务决定，而非全部堆入 🔶 全量 messages 追加
无选择性上下文管理		 缺：上下文窗口管理、任务状态外部化、Progress 文件、跨会话状态
T4: 独立于模型的控制？ 权限、预算、测试、策略门能独立阻断或确认执行 ❌ 无独立控制
Prompt 中的约束全靠模型自觉		 缺：Policy Gate、工具白名单、token 预算硬限制、独立验收、审批门
评估结论
当前项目拥有完整的 T1（ReAct Loop）和部分的 T2（基础工具），但在 T3（任务感知上下文）和 T4（独立控制）上存在明显缺口。这也是 Phase 3-6 需要逐步补齐的 Harness 能力。

11.3 Harness 分层设计与目标架构

参考 Harness Engineering 概念导论的六层模型（L0-L5），映射到本项目：

┌──────────────────────────────────────────────────────────────┐ │ L5 治理与组织 (Phase 6) │ │ 身份认证 · 多租户 · 成本审计 · 审批流程 · 平台模板 │ ├──────────────────────────────────────────────────────────────┤ │ L4 质量与交付 (Phase 4-6) │ │ Event Logs · Trace · 独立验收 · Eval 回归 · Policy Gate │ ├──────────────────────────────────────────────────────────────┤ │ L3 任务运行与服务 (Phase 1 ✅ → Phase 3-5) │ │ ReAct Loop ✅ · 编排 · 状态管理 · 记忆 · 恢复 · │ │ Task Contract · Progress · Checkpoint · Heartbeat │ ├──────────────────────────────────────────────────────────────┤ │ L2 行动与环境 (Phase 1 ✅ → Phase 5) │ │ Tools (3 builtins) ✅ · MCP · Workspace · Sandbox │ ├──────────────────────────────────────────────────────────────┤ │ L1 指导与上下文 (Phase 1 ✅) │ │ System Prompt ✅ · Agent Config ✅ · Skills · RAG │ ├──────────────────────────────────────────────────────────────┤ │ L0 模型 (Phase 1 ✅) │ │ OpenAI-compatible API ✅ · SSE Streaming ✅ │ └──────────────────────────────────────────────────────────────┘
设计原则
Harness 的厚度跟任务风险匹配。简单对话只需要薄薄一层（消息历史 + 超时 + 日志）；跨系统写入、运行数小时、能取消和恢复的 Agent 任务，需要明显更厚的运行与控制层。本项目的每个 Phase 逐渐增厚 Harness，而非一次性全量实现。

11.4 任务契约（Task Contract）

核心思想来自 Anthropic 长任务案例：把"什么算完成"从人脑里搬到系统对象里。任务开始前，明确声明范围、验收标准、预算、权限和升级条件。

TaskContract 结构体

// TODO: Phase 3 — TaskContract 定义
// 位置: internal/harness/contract.go

type TaskContract struct {
    // 任务身份
    TaskID    string   `json:"task_id"`
    ProjectID string   `json:"project_id"`  // 项目隔离

    // 目标与范围
    Goal        string   `json:"goal"`         // 高层目标（一句话）
    Scope       string   `json:"scope"`        // 范围边界（明确不做什么）
    Deliverables []string `json:"deliverables"` // 可交付物清单

    // 验收标准（结构化，不只是自然语言）
    AcceptanceCriteria []AcceptanceCriterion `json:"acceptance_criteria"`
    // 示例: { type: "test_pass", target: "go test ./...", expected: "PASS" }
    //       { type: "file_exists", target: "output/report.md" }
    //       { type: "shell_exit_zero", target: "docker build ." }

    // 资源预算
    Budget TaskBudget `json:"budget"`
    // MaxTokens: 50000, MaxCostUSD: 0.50, MaxDurationSec: 300

    // 权限边界
    Permissions TaskPermissions `json:"permissions"`
    // AllowedTools: ["read_file", "write_file", "run_shell"]
    // FileScope: "storage/{task_id}/"  (只允许在这个目录下写文件)
    // NetworkAccess: false
    // RequireApprovalFor: ["run_shell:rm", "run_shell:git push"]

    // 升级策略
    Escalation EscalationPolicy `json:"escalation"`
    // OnBudgetExceeded: "pause_and_ask_user"
    // OnTimeout: "checkpoint_and_exit"
    // On3ConsecutiveFailures: "escalate_to_human"
}

验收标准类型

类型说明示例
test_pass运行指定测试套件并通过go test ./... → PASS
file_exists指定文件存在且非空output/report.md 存在 > 100 bytes
shell_exit_zeroShell 命令退出码为 0docker build . → exit 0
json_schema输出符合 JSON Schema结构化输出校验
human_review需要人工确认修改 >3 个文件时触发
与 Prompt 的区别
Prompt 是对模型的"建议"；TaskContract 是 Harness 的"强制约束"。预算超了就是超了，不管模型怎么说。文件只允许写在 storage/{task_id}/ 下，路径遍历直接拦截。

11.5 Progress 文件与外部状态

参考 Anthropic 的 claude-progress.txt 和 OpenAI 的 execution plan：任务状态不能只存在于聊天上下文里，必须外部化到可见、可版本化、可验证的文件中。

Progress 文件设计

// TODO: Phase 3 — Progress 文件管理
// 位置: internal/harness/progress.go

// 每个 Task 在 storage/{task_id}/ 下维护 progress.json
// 引擎在关键节点自动写入，新会话启动时首先读取

type TaskProgress struct {
    TaskID     string           `json:"task_id"`
    Status     string           `json:"status"`  // planning | executing | verifying | completed
    UpdatedAt  time.Time        `json:"updated_at"`

    // 当前执行位置
    CurrentStep    int    `json:"current_step"`
    CurrentSubGoal string `json:"current_sub_goal"`

    // 已完成 / 待完成
    CompletedItems []ProgressItem `json:"completed_items"`
    PendingItems   []ProgressItem `json:"pending_items"`

    // 决策记录（为什么这么做，而非只记录做了什么）
    DecisionLog []Decision `json:"decision_log"`
    // 示例: { decision: "使用 Docker 多阶段构建", reason: "减少最终镜像体积",
    //         alternatives: ["单阶段构建", "distroless"], timestamp: "..." }

    // 当前环境状态
    EnvironmentState EnvironmentSnapshot `json:"environment_state"`
    // GitCommit: "abc1234", LastTestRun: "PASS", PortsInUse: [8080, 3000]
}

// ProgressItem 是功能清单中的一项
// 参考 Anthropic: 初始全部标记为 failing，实际验证通过后才改为 passing
type ProgressItem struct {
    ID          string `json:"id"`
    Description string `json:"description"`
    Status      string `json:"status"` // pending | in_progress | passing | failing
    VerifiedBy  string `json:"verified_by"` // 验收方式: "go test ./..." | "manual" | "file_exists:..."
}

工作流

Task 启动 │ ├─ 1. 读取 progress.json（如果存在 → 恢复；不存在 → 新建） ├─ 2. 读取 Git 状态 + 上次基线测试结果 ├─ 3. 识别当前 SubGoal 和下一步 ├─ 4. 执行一个 SubGoal（一次只做一个功能） ├─ 5. 验证 → 更新 ProgressItem status ├─ 6. 写回 progress.json + Git commit └─ 循环直到全部 passing
关键设计决策
一次只做一个功能。Anthropic 实验表明 Agent 一次做太多会导致上下文耗尽时留下半成品。Progress 文件强制分步、强制验证、强制交接 —— 新会话不需要从聊天摘要猜测上一会话做了什么。

11.6 Policy Gate — 独立于模型的约束层

这是 T4 的核心实现。Policy Gate 在工具执行之前拦截，而非执行之后补救。它是确定性的 Go 代码，不依赖模型是否"听话"。

PolicyGate 接口

// TODO: Phase 4 — Policy Gate
// 位置: internal/harness/policy.go

// PolicyGate 是工具执行前的安全检查层。
// 所有检查都是确定性的 Go 代码 —— 不依赖 LLM 自觉。
// 任何一条规则返回 false，工具调用即被拒绝，Engine 收到 ErrBlockedByPolicy。
type PolicyGate interface {
    // Allow 判断是否允许执行指定的工具调用。
    // 返回 nil 表示允许；返回 error 表示拒绝（含拒绝原因）。
    Allow(taskID string, toolName string, args map[string]any) error
}

// 内置规则链（Chain of Responsibility）
type PolicyChain struct {
    rules []PolicyRule
}

// PolicyRule 是单条策略规则
type PolicyRule interface {
    Name() string
    Check(taskID string, toolName string, args map[string]any) error
}

// 预置规则示例：
// - PathTraversalRule: 拒绝 "../" 路径遍历
// - FileScopeRule: 只允许在 storage/{task_id}/ 下写文件
// - DangerousCommandRule: 拒绝 rm -rf /、fork bomb、curl | bash
// - TokenBudgetRule: 超过 token 预算硬限制时拒绝继续
// - ToolWhitelistRule: 只允许 TaskContract 中声明的工具
// - ApprovalRule: 高风险操作（如 git push）需要前端确认

拦住什么？

规则拦截条件Phase
PathTraversalRule路径包含 .. 或绝对路径越界3 ✅ 已有基础
FileScopeRule写入 storage/{task_id}/ 之外的路径3
DangerousCommandRulerm -rf /、fork()、curl | sh 等4
TokenBudgetRule累计 token 超过 TaskContract 预算4
ToolWhitelistRule使用了 TaskContract 未声明的工具4
ApprovalRule高风险操作 → 前端弹窗确认5
设计铁律
Policy Gate 的逻辑必须全部是确定性的 Go 代码。不要把"判断是否危险"也交给模型 —— 那等于把监狱的钥匙交给了囚犯。

11.7 Checkpoint / Recovery — 任务可恢复性

Agent 任务可能运行数分钟甚至数小时。进程崩溃、网络中断、API 限流都可能打断执行。Checkpoint 机制确保任务可以从中断点恢复，而非从头开始。

// TODO: Phase 4 — Checkpoint 机制
// 位置: internal/harness/checkpoint.go

type Checkpoint struct {
    TaskID      string          `json:"task_id"`
    CreatedAt   time.Time       `json:"created_at"`
    StepIndex   int             `json:"step_index"`

    // 完整对话历史（用于恢复 LLM 上下文）
    Messages    []llm.Message   `json:"messages"`

    // 当前 Progress 快照
    Progress    TaskProgress    `json:"progress"`

    // 文件系统快照（可选，Phase 5+）
    // WorkspaceSnapshot string  `json:"workspace_snapshot"`
}

// CheckpointManager 管理 checkpoint 的创建和恢复
type CheckpointManager interface {
    Save(ctx context.Context, cp Checkpoint) error
    LoadLatest(taskID string) (*Checkpoint, error)
    // 恢复流程:
    // 1. 从 SQLite 加载最新 checkpoint
    // 2. 重建 Engine 的 messages 切片
    // 3. 从 progress.json 恢复当前 SubGoal
    // 4. 重新执行被中断的 step
    Recover(taskID string, engine *Engine) error
}
触发时机
  • 每个 SubGoal 完成后：自动创建 checkpoint
  • 工具执行失败前：保存当前状态再退出
  • Token 预算耗尽前：提前 10% 创建 checkpoint
  • 定期心跳：每 N 步自动 checkpoint（Phase 4+）

11.8 自进化长期记忆 — 四层模型

参考 Hermes Agent 的记忆架构和 CoALA 认知模型，将长期记忆分为四层。核心洞察：记忆是治理问题，不只是存储问题。RAG 只覆盖"召回"，写入、修正、遗忘这三个动作决定"这条记录未来该不该影响行为"。

四层记忆总览

层放什么默认进 Prompt?存储治理方式
Working Memory 当前任务必要信息：TaskContract + Stable Rules + 相关 Episode 摘要 ✅ 是 内存 + task_brief.json 任务开始前构建，任务结束后丢弃
Raw Episodic 原始事件流：对话、工具调用、工具结果 ❌ 否 SQLite conversations 表 只追加、不修饰、不删除
Consolidated Episodic 任务摘要、关键决策、失败原因、候选经验 🔶 候选 SQLite memories 表 心跳整理 + 模型抽取 + 证据链接
Semantic / Policy 稳定规则、用户偏好、长期事实、项目经验 ✅ 高优先 SQLite memories 表 少而稳定，每条必须有晋升理由和证据。宁可漏记，不可记错

Hermes Agent 映射

Hermes Agent 结构本项目对应落地方式
MEMORY.md / USER.mdSemantic / Policy 层memories 表，type=preference|rule|fact，带 evidence + promotion_reason
SQLite + session_searchRaw Episodic 层现有 conversations 表，按 task_id 查询
会话结束抽取 / 后台 reviewConsolidated + Semantic 晋升Heartbeat 定时触发整理管线
Memory Provider可选向量检索扩展Phase 6 向量检索（LanceDB / ChromaDB），不作为主存储

数据库 Schema 扩展

-- TODO: Phase 4 — memories 表 + memory_links 表
-- 位置: pkg/db/database.go (createTables 中追加)

CREATE TABLE IF NOT EXISTS memories (
    id           TEXT PRIMARY KEY,
    project_id   TEXT NOT NULL,     -- 项目隔离（不同 project 的记忆不互通）
    type         TEXT NOT NULL,     -- preference | rule | fact | lesson | reflection
    tier         TEXT NOT NULL,     -- consolidated | semantic
    content      TEXT NOT NULL,     -- 记忆内容（脱离上下文也能读懂的一句话）
    embedding    BLOB,              -- 向量（Phase 6+，用于语义检索）
    confidence   REAL DEFAULT 1.0,  -- 置信度 0-1
    status       TEXT DEFAULT 'active', -- active | obsolete | cold | invalid

    -- 证据链（每条记忆必须能追溯到来源）
    source_task_ids  JSON,          -- 来源任务 ID 列表
    source_event_ids JSON,          -- 来源事件 ID 列表
    promotion_reason TEXT,          -- 晋升理由: repeated_across_sessions | tool_failure_evidence | explicit_user_instruction

    -- 生命周期
    access_count    INT DEFAULT 0,
    last_accessed   DATETIME,
    last_reviewed   DATETIME,       -- 上次审查时间（用于过期检测）
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 记忆关联表（记忆之间的引用关系）
CREATE TABLE IF NOT EXISTS memory_links (
    source_id  TEXT NOT NULL,
    target_id  TEXT NOT NULL,
    relation   TEXT NOT NULL,  -- supplements | contradicts | relates_to | replaces
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (source_id, target_id),
    FOREIGN KEY (source_id) REFERENCES memories(id),
    FOREIGN KEY (target_id) REFERENCES memories(id)
);

11.9 记忆抽取管线

四层记忆的抽取不是一次性完成的，而是通过一条异步管线逐步压缩和晋升。核心分工原则：

分工原则
理解自然语言的判断交给模型（哪句话值得记？两条候选是不是同一件事？用户是否显式说"以后都这样"？）
可核对的结构化判断留在代码（工具返回的 status 是不是 failed？一条候选被几个 task 独立支持？access_count 是否过低？）

抽取管线流程

Raw Events (conversations 表) │ │ Step 1: 模型抽取 (Consolidation Agent) │ 输入: 单个 Task 的全部 conversation 记录 │ 输出: episode_summary + memory_items[] │ 每项 memory_item 带 type、statement、supported_event_ids、durable 标记 │ ▼ Candidate Memories (memories 表, tier=consolidated) │ │ Step 2: 模型合并同义候选 (Dedup Agent) │ 输入: 同 project 下的所有 candidate │ 输出: merged_memories[]（同义合并，不发明新内容） │ │ Step 3: 代码按证据决定晋升 (Promotion Gate) │ 三条晋升通道: │ ✓ repeated_across_sessions: 被 ≥2 个独立 task 支持 │ ✓ tool_failure_evidence: 关联的 event 中有 tool status=failed │ ✓ explicit_user_instruction: 用户显式说"以后都这样"/durable=true │ 不满足任一条 → 留在 candidate，等下次证据 │ ▼ Semantic Rules (memories 表, tier=semantic) │ │ Step 4: 定期审查 (Prune Agent) │ 超过 30 天未 access → status=cold │ 被更新版本覆盖 → status=obsolete │ confidence 持续下降 → 标记待审查 │ ▼ Working Memory (task_brief.json) │ 新任务启动时按 project_id + 关键词 + 向量相似度召回 │ 注入 System Prompt 或作为 context 前置

三条晋升通道

通道触发条件代码判断示例
repeated_across_sessions 同一经验被 ≥2 个独立 task 支持 COUNT(DISTINCT source_task_id) >= 2 "远距离景点单独占一整天"——北京和西安都犯了同样的错
tool_failure_evidence 关联的 event 中有 tool status=failed event_index.status == "failed" "route_checker 报告跨区移动不可行"——工具返回了确凿的失败证据
explicit_user_instruction 用户显式说"以后都这样"或 durable=true memory_item.durable == true "以后输出行程时附上检查结论"——用户明确要求长期生效
晋升门槛宁紧勿松
Semantic 层进 Prompt 的优先级最高。进去一条错的，以后每个任务都要为它买单。只出现过一次、没有失败证据、用户也没说"以后都这样"的信息，留在 consolidated 层等下次证据，而不是急着变成规则。

11.10 心跳整理（Heartbeat Consolidation）

记忆整理不适合放在对话进行中做——用户还在等回复，没人愿意每轮多等几秒等 Agent "记笔记"。心跳机制是一个后台定时任务，在系统空闲时扫描新事件、触发抽取管线、更新 checkpoint。

// 自适应间隔
func (h *Heartbeat) adaptiveInterval(newEventCount int) time.Duration {
    switch {
    case newEventCount >= 10: return 60 * time.Second   // 活跃
    case newEventCount >= 1:  return 5 * time.Minute    // 有活动
    default:                  return 15 * time.Minute   // 空闲
    }
}

心跳安全边界

自动整理不能变成自动污染
心跳是后台进程，它读到的内容不只有用户对话——邮件、消息、网页、仓库都可能流进来。后台写记忆这条通道如果没有治理，会出现安全问题（参考 Mind Your HEARTBEAT! 论文，arXiv 2603.23064）。

防御措施：
  • 每条记忆必须有 source_event_ids（可追溯）
  • 每条记忆必须有 promotion_reason（可解释）
  • 低置信度记忆不进入 Semantic 层
  • 支持记忆回滚（通过 memory_links 的 replaces 关系）
  • 定期审查：超过 30 天未 access 的 Semantic 规则自动降级

11.11 记忆召回 — 新任务唤起

新任务启动时，不是把所有历史灌进上下文，而是按优先级构建 Working Memory：

新任务: "帮我在项目中加一个 API 认证中间件" │ ├─ Step 1: 项目隔离 │ project_id = 当前项目 → 只看本项目的记忆 │ ├─ Step 2: Semantic 规则（整批带入，优先级最高） │ SELECT * FROM memories WHERE tier='semantic' AND project_id=? │ → "项目使用 gorilla/mux 作为路由框架" │ → ".env 中存储 API Key" │ → "所有中间件放在 internal/middleware/ 目录" │ ├─ Step 3: Consolidated Episodes（按相关性选取 Top 3） │ 关键词匹配 + (Phase 6: 向量相似度) → 排序 │ → 上次添加 logging 中间件的 task 摘要 │ → 上次修改路由配置的 task 摘要 │ ├─ Step 4: 打包为 task_brief.json │ { │ "task": "...", │ "stable_rules": [...], │ "related_episodes": [...] │ } │ └─ Step 5: 注入 System Prompt "以下是从以往任务中总结的稳定规则和相关经验: ..."

// BuildWorkingMemory 为新任务构建 working memory context
func (r *MemoryRecall) BuildWorkingMemory(projectID, taskGoal string) (*WorkingMemory, error) {
    wm := &WorkingMemory{TaskGoal: taskGoal}

    // 1. 加载 Semantic 规则（整批）
    wm.StableRules = r.loadSemanticRules(projectID)

    // 2. 关键词匹配 + 召回相关 Episode（Top N）
    wm.RelatedEpisodes = r.recallEpisodes(projectID, taskGoal, 3)

    // 3. 构建 task_brief.json
    wm.BuildBrief()

    return wm, nil
}
