12.1 概述：Skill 与 Workflow 定位

Claude Code 提供了两种本质不同的扩展机制来增强 Agent 的能力。它们解决的是不同层次的问题：

┌──────────────────────────────────────────────────────────────┐ │ Claude Code 扩展体系 │ │ │ │ ┌─────────────────────────┐ ┌─────────────────────────────┐│ │ │ Skill │ │ Workflow ││ │ │ │ │ ││ │ │ "教一个 Agent 怎么做" │ │ "编排多个 Agent 协作完成" ││ │ │ │ │ ││ │ │ • 单 Agent 知识注入 │ │ • 多 Agent 编排引擎 ││ │ │ • Markdown 文件 │ │ • JavaScript 脚本 ││ │ │ • 关键词触发 │ │ • 显式调用 (/workflow) ││ │ │ • AI 自由遵循 │ │ • 确定性执行 ││ │ │ • 渐进式信息披露 │ │ • 并行/流水线模式 ││ │ │ • 无状态 │ │ • 有状态 (预算/缓存/恢复) ││ │ └─────────────────────────┘ └─────────────────────────────┘│ │ │ │ 类比: Skill = 给工匠一本手册 Workflow = 给工头一张施工图 │ └──────────────────────────────────────────────────────────────┘

核心区别一句话

维度SkillWorkflow
本质 单 Agent 的知识扩展——往 Agent 的上下文中注入领域知识和操作指南 多 Agent 的编排引擎——定义多个 Agent 如何分工、协作、传递结果
类比 给一个聪明人一本详细的 SOP 手册，他按手册自由发挥 给一个工头一份施工蓝图，他指挥多个工人按图施工，每个步骤有明确的质量检查
解决的问题 "Agent 不知道这个领域的知识 / 不知道怎么做这个特定任务" "单个 Agent 做不完这个任务 / 需要多视角分析 / 需要严格的质量门控"
设计哲学
Skill 和 Workflow 不是互斥的，而是互补的。一个 Workflow 的每个 Phase 都可以使用 Skill 来增强各个 Agent 的能力。 选择 Skill 还是 Workflow，取决于任务的结构化程度和是否需要多 Agent 协作。

12.2 Skill 深度解析

12.2.1 SKILL.md 的解剖结构

Skill 的核心是一个 Markdown 文件，由两部分组成：YAML 前置元数据 (frontmatter) 和 Markdown 正文。

---
name: backend-patterns
description: |
  Backend architecture patterns, API design,
  database optimization, and server-side best
  practices for Node.js, Express, and Next.js.
origin: ECC
---

# Backend Development Patterns

Backend architecture patterns and best practices...

## When to Activate
- Designing REST or GraphQL API endpoints
- Implementing repository, service, or controller layers

## API Design Patterns
### RESTful API Structure
...

Frontmatter 字段详解

字段必需说明示例
name 是 Skill 的唯一标识符，用于系统内部引用 backend-patterns
description 是 触发匹配的核心字段。系统将用户输入与此字段做关键词匹配，决定是否激活该 Skill Backend architecture patterns, API design, database optimization...
trigger 否 显式触发命令（如 /graphify），用户输入此命令时直接激活 /graphify
origin 否 来源标识，用于追踪 Skill 的出处 ECC (Everything Claude Code)

12.2.2 触发机制：基于描述的关键词匹配

Skill 的触发不是显式的函数调用，而是基于语义匹配的自动激活。系统的工作流程如下：

用户输入: "帮我设计一个 REST API 的认证中间件" │ ▼ 系统扫描所有已安装 Skill 的 description 字段 │ ├─ backend-patterns: "Backend architecture patterns, API design..." │ → 关键词匹配: "REST API" + "设计" + "认证" + "中间件" → ✅ 匹配 │ ├─ frontend-patterns: "Frontend architecture patterns, UI..." │ → 关键词匹配: 无相关词 → ❌ 不匹配 │ └─ coding-standards: "Coding standards and best practices..." → 关键词匹配: 弱相关 → 🔶 部分匹配（低优先级） │ ▼ 匹配的 Skill 被注入到 Agent 的 System Prompt 中 → Agent 在生成回复时参考 Skill 中的知识和指令
关键理解
Skill 的触发是概率性的、非确定性的。系统通过关键词匹配判断是否激活，但最终 Agent 是否严格遵循 Skill 中的指令，取决于模型自身的判断。这与 Workflow 的确定性执行形成鲜明对比。

description 编写最佳实践

  • 覆盖所有触发场景的关键词：如果 Skill 是关于"Go 后端开发"，description 应包含 "Go", "golang", "backend", "API", "server", "database" 等关键词
  • 使用自然语言描述：不要堆砌关键词，而是写一句通顺的话，自然地包含关键词
  • 明确 When to Activate：在正文的 "When to Activate" 部分，列出所有应该触发此 Skill 的具体场景

12.2.3 渐进式信息披露 (Progressive Disclosure)

Skill 采用三层信息结构，系统按需加载，避免一次性将所有内容灌入上下文窗口：

Layer 1: 元数据 (Metadata) ├─ name, description, trigger ├─ 作用: 匹配阶段，判断是否激活 └─ Token 成本: ~50 tokens Layer 2: 正文 (Body) ├─ Markdown 主体内容 ├─ 作用: 激活后注入 Agent 上下文 └─ Token 成本: ~500-3000 tokens Layer 3: 捆绑资源 (Bundled Resources) ├─ scripts/ — 可执行脚本 ├─ references/ — 参考文档（按需加载） └─ assets/ — 模板、图片等静态资源 └─ Token 成本: 仅在 Agent 显式引用时加载
设计意图
渐进式披露确保 上下文窗口不被无关内容污染。Layer 1 用于匹配（便宜），Layer 2 在匹配成功后注入（中等成本），Layer 3 仅在需要时由 Agent 主动加载（按需付费）。这保证了 Skill 系统可以扩展到数百个 Skill 而不显著增加每次对话的 token 开销。

12.2.5 何时使用 Skill

场景示例原因
领域知识注入 后端架构模式、前端最佳实践、数据库优化 Agent 需要特定领域的专业知识，但不需要多个 Agent 协作
固定流程的 SOP Bug 修复流程、代码审查清单、部署检查表 流程是固定的步骤序列，单个 Agent 可以执行
编码规范 Go 编码规范、Vue 组件风格指南、命名约定 规范是约束性知识，需要注入到每次对话中
工具使用指南 Docker 使用模式、Git 工作流、CI/CD 配置 Agent 需要知道如何正确使用工具，但执行是单 Agent 的
项目特定约定 CLAUDE.md 中的项目指令、目录结构约定 每个项目有自己的规则，需要自动注入
Skill 的局限
Skill 不能做的事：
  • 不能编排多个 Agent 并行工作
  • 不能强制执行步骤（Agent 可以自由选择是否遵循）
  • 不能追踪状态（无状态，每次对话独立）
  • 不能做质量门控（没有"通过/不通过"的强制判断）
  • 不能做预算管理（无法限制 token 消耗或执行时间）

12.3 Workflow 深度解析

12.3.1 定义：多 Agent 编排引擎

Workflow 是一个确定性的多 Agent 编排脚本，用 JavaScript 编写。它定义了多个 Agent 的分工、协作模式、数据传递路径和质量检查点。与 Skill 的"建议"性质不同，Workflow 是强制执行的——每个 Phase 必须完成，每个质量门必须通过，否则流程终止。

Workflow 脚本 (JavaScript) │ ├─ export const meta = { name, description, budget } │ ├─ phase("Research", async () => { ... }) │ ├─ agent("analyzer", { prompt: "...", model: "sonnet" }) │ └─ log("Analysis complete", { findings: [...] }) │ ├─ phase("Ideation", async () => { │ │ parallel([ │ │ agent("codex", { prompt: "..." }), │ │ agent("gemini", { prompt: "..." }), │ │ ]) │ │}) │ ├─ phase("Plan", async () => { │ │ pipeline([ │ │ agent("architect", { prompt: "..." }), │ │ agent("reviewer", { prompt: "..." }), │ │ ]) │ │}) │ ├─ phase("Execute", async () => { ... }) │ └─ phase("Review", async () => { │ qualityGate({ score: 7, ... }) │})

12.3.2 脚本格式：meta 块 + 阶段定义

每个 Workflow 脚本是一个 JavaScript 模块，必须导出 meta 对象：

// 文件名: code-review.workflow.js

export const meta = {
    name: "code-review",
    description: "Multi-model code review with adversarial verification",
    budget: {
        total: 100000,    // 总 token 预算
        perAgent: 30000,   // 每个 Agent 的 token 预算
        maxTime: 600,      // 最大执行时间（秒）
    },
    resume: true,          // 支持中断恢复
    cache: "phase",        // 缓存策略: "phase" | "agent" | "none"
};

meta 字段详解

字段类型说明
namestringWorkflow 的唯一名称
descriptionstring描述用途，用于 /workflow list 展示
budget.totalnumber整个 Workflow 的 token 预算上限。超限时自动暂停，等待用户确认
budget.perAgentnumber每个 Agent 实例的 token 预算。防止单个 Agent 消耗过多
budget.maxTimenumber最大执行时间（秒），超时后自动创建 checkpoint 并退出
resumeboolean是否启用中断恢复。启用后，Workflow 在关键节点自动保存状态
cachestring缓存策略："phase" 缓存每个 Phase 的输出，"agent" 缓存每个 Agent 的输出，"none" 不缓存

12.3.3 核心 API

agent() — 创建单个 Agent 任务

// 最基本的用法：创建一个 Agent 并等待其完成
const result = await agent({
    name: "code-analyzer",
    prompt: "分析以下代码的安全漏洞：...",
    model: "sonnet",           // 模型选择
    tools: ["read_file", "grep"],  // 允许的工具
    maxSteps: 15,               // 最大步数
    context: { ... },            // 注入的上下文数据
});

// result 包含:
// { output: string, tokens: { input, output }, steps: number, errors: [] }

parallel() — 并行执行多个 Agent

// 同时启动多个 Agent，等待全部完成
const [backendResult, frontendResult] = await parallel([
    agent({ name: "backend-review", prompt: "审查后端代码...", model: "sonnet" }),
    agent({ name: "frontend-review", prompt: "审查前端代码...", model: "haiku" }),
]);

// parallel() 的特点:
// - 所有 Agent 同时启动，不等待彼此
// - 全部完成后才返回结果
// - 任何一个失败，整个 parallel 块失败
// - 适用于：多视角分析、独立审查、批量处理

pipeline() — 流水线顺序执行

// 多个 Agent 串联，前一个的输出作为后一个的输入
const finalResult = await pipeline([
    // Stage 1: 分析
    agent({ name: "analyzer", prompt: "分析代码库结构..." }),

    // Stage 2: 基于分析结果进行规划
    ({ output: analysis }) => agent({
        name: "planner",
        prompt: `基于以下分析制定重构计划：\n${analysis}`,
    }),

    // Stage 3: 基于计划执行
    ({ output: plan }) => agent({
        name: "executor",
        prompt: `执行以下重构计划：\n${plan}`,
    }),
]);

// pipeline() 的特点:
// - 严格按顺序执行
// - 前一个的输出自动传递给下一个
// - 任何一个失败，pipeline 中断
// - 适用于：分析→规划→执行 的线性流程

phase() — 定义执行阶段

// phase() 将 Workflow 分为命名阶段，便于追踪和恢复
await phase("Research", async (ctx) => {
    ctx.log("开始研究阶段");

    const findings = await agent({
        name: "researcher",
        prompt: "研究项目的技术债务...",
    });

    // ctx.data 在阶段间持久化
    ctx.data.findings = findings.output;

    ctx.log("研究阶段完成", { tokens: findings.tokens });
});

// phase() 的特点:
// - 每个 phase 是独立的恢复点
// - ctx.data 在 phase 间传递数据
// - ctx.log() 输出结构化日志
// - 如果 resume: true，中断后从最后一个完成的 phase 恢复

log() — 结构化日志

// log() 输出结构化日志，用于追踪和调试
log("Agent 完成分析", {
    agent: "code-analyzer",
    tokens: { input: 5000, output: 1200 },
    duration: "12.5s",
    findings: ["SQL injection risk", "XSS vulnerability"],
});

// 日志级别:
log.info("信息");
log.warn("警告");
log.error("错误");

12.3.4 预算系统 (Budget System)

Workflow 的预算系统是强制性的——不是建议，而是硬限制。它在三个层面控制资源消耗：

层面控制项超限行为
Workflow 级别 budget.total — 整个 Workflow 的 token 上限 暂停执行，显示预算使用报告，等待用户确认继续或终止
Agent 级别 budget.perAgent — 每个 Agent 的 token 上限 Agent 被强制终止，输出已生成的部分结果，错误信息记录到日志
时间级别 budget.maxTime — 最大执行时间 自动创建 checkpoint，保存当前状态，优雅退出 
// 运行时查询预算状态
const status = budget.status();
// { total: 100000, spent: 45000, remaining: 55000, agentCount: 3 }

// 在关键节点检查预算
if (budget.remaining() < 10000) {
    log.warn("预算即将耗尽，创建 checkpoint");
    await checkpoint.save();
}

12.3.5 恢复/缓存机制 (Resume & Caching)

Workflow 支持中断恢复和结果缓存，这是它与 Skill 的关键区别之一：

Workflow 执行流程与恢复机制: Phase 1: Research ✅ 已完成 → 结果缓存 Phase 2: Ideation ✅ 已完成 → 结果缓存 Phase 3: Plan 🔄 执行中 → 中断！ │ ▼ 用户重新运行: /workflow resume code-review │ ├─ 检查缓存: Phase 1 ✅ → 跳过 ├─ 检查缓存: Phase 2 ✅ → 跳过 └─ Phase 3: 从 checkpoint 恢复，继续执行 
// 缓存策略配置
export const meta = {
    cache: "phase",  // 缓存每个 phase 的结果
    // "agent" — 缓存每个 agent 的结果（粒度更细）
    // "none"  — 不缓存（每次重新执行）
};

// 手动创建 checkpoint
await checkpoint.save({
    phase: "Plan",
    data: ctx.data,
    timestamp: Date.now(),
});

// 恢复时自动加载
// checkpoint.load() 返回最近一次保存的状态

12.3.6 Pipeline vs Parallel：选择指南

模式适用场景不适用场景
Pipeline
(顺序执行)		 • 有数据依赖的流程（分析→规划→执行）
• 需要逐步精炼的任务（草稿→审查→终稿）
• 质量门控流程（执行→检查→通过/驳回） 		• 独立任务（浪费并行能力）
• 快速探索（等待时间过长）
Parallel
(并行执行)		 • 多视角分析（后端+前端+安全 同时审查）
• 批量处理（多个文件独立分析）
• 对抗性验证（两个 Agent 独立验证同一个结果） 		• 有依赖关系的任务（B 需要 A 的输出）
• 需要全局一致性的任务（容易产生冲突）

12.3.7 常见 Workflow 模式

模式 1：对抗性验证 (Adversarial Verify)

// 两个 Agent 独立分析，结果交叉验证
phase("Verify", async (ctx) => {
    const [result1, result2] = await parallel([
        agent({ name: "verifier-a", prompt: "独立验证以下代码...", model: "sonnet" }),
        agent({ name: "verifier-b", prompt: "独立验证以下代码...", model: "haiku" }),
    ]);

    // 交叉比对，找出不一致
    const conflicts = findConflicts(result1.output, result2.output);
    if (conflicts.length > 0) {
        log.warn("发现不一致", { conflicts });
        // 可以启动第三个 Agent 做仲裁
    }
});

模式 2：循环直到合格 (Loop Until Dry)

// 反复执行直到没有新发现
phase("Review", async (ctx) => {
    let iteration = 0;
    let foundIssues = true;

    while (foundIssues && iteration < 5) {
        const result = await agent({
            name: "reviewer",
            prompt: `审查代码（第 ${iteration + 1} 轮）：\n${ctx.data.code}`,
        });

        foundIssues = result.output.includes("ISSUE");
        if (foundIssues) {
            // 自动修复发现的问题
            ctx.data.code = await autoFix(result.output);
        }
        iteration++;
    }
});

模式 3：多阶段流水线 (Multi-Phase Pipeline)

// 完整的多阶段开发流程
phase("Research", async (ctx) => {
    ctx.data.context = await agent({ name: "researcher", prompt: "研究..." });
});

phase("Ideation", async (ctx) => {
    const [backend, frontend] = await parallel([
        agent({ name: "backend-ideas", prompt: `基于 ${ctx.data.context} 提出后端方案` }),
        agent({ name: "frontend-ideas", prompt: `基于 ${ctx.data.context} 提出前端方案` }),
    ]);
    ctx.data.ideas = { backend, frontend };
});

phase("Plan", async (ctx) => {
    ctx.data.plan = await agent({
        name: "planner",
        prompt: `综合方案制定计划：\n${JSON.stringify(ctx.data.ideas)}`,
    });
});

phase("Execute", async (ctx) => {
    await agent({ name: "executor", prompt: `执行计划：\n${ctx.data.plan}` });
});

phase("Review", async (ctx) => {
    const score = await qualityGate();
    if (score < 7) throw new Error("质量不达标");
});

12.3.8 何时使用 Workflow

场景示例原因
大规模代码审查 全仓库安全审计、跨模块重构审查 需要多个 Agent 并行分析不同模块，结果交叉验证
多视角分析 后端+前端+安全+性能 四维分析 不同视角需要不同的 Agent 和不同的模型
批量迁移 从 REST 迁移到 GraphQL、从 JS 迁移到 TS 大量文件需要独立处理，并行执行显著加速
质量门控流程 代码生成→审查→测试→部署 的 CI/CD 流水线 需要强制性的质量检查点，不合格必须终止
对抗性验证 两个 Agent 独立实现同一功能，结果对比 需要确定性的一致性检查，不能依赖模型自觉
多 Agent 测试 Agent A 生成测试用例，Agent B 执行测试，Agent C 分析结果 需要严格的阶段依赖和数据传递

12.4 对比表：十一维全维度对比

维度SkillWorkflow
本质 单 Agent 知识注入 多 Agent 编排引擎
文件格式 Markdown (YAML frontmatter + body) JavaScript (可执行脚本)
触发方式 自动（关键词匹配 description） 显式（用户执行 /workflow 命令）
确定性 非确定性 —— Agent 自由选择是否遵循 确定性 —— 脚本定义的步骤强制执行
Agent 数量 1 个（当前对话的 Agent） N 个（可编排多个 Agent 并行/流水线）
状态管理 无状态 —— 每次对话独立 有状态 —— 预算追踪、检查点、缓存、恢复
预算控制 无 —— 不限制 token 或时间 有 —— 三层预算（Workflow/Agent/时间）
质量门控 无 —— 无法强制执行质量检查 有 —— 每个 Phase 可以设置评分门槛
执行模型 同步 —— 注入到当前对话的上下文中 异步 —— 后台执行，支持并行和流水线
信息加载 渐进式披露（元数据→正文→资源） 全量加载（脚本在启动时全部解析）
复杂度 低 —— 编写 Markdown 即可 高 —— 需要编写 JavaScript 编排逻辑
适用任务规模 小到中型 —— 单次对话能完成的任务 中到大型 —— 需要多轮、多 Agent 协作的任务
可恢复性 不可恢复 —— 中断后需重新开始 可恢复 —— 从 checkpoint 继续执行
学习曲线 低 —— 会写 Markdown 就会写 Skill 中高 —— 需要理解编排概念和异步编程

12.5 决策指南：何时用 Skill，何时用 Workflow

当你面对一个任务时，按以下决策树选择：

Q1: 这个任务是否需要多个 Agent 协作？

├─ 是 → Workflow（例如：后端+前端同时审查、多视角分析）

└─ 否 → 继续 Q2

Q2: 这个任务是否需要强制性的质量检查/预算控制？

├─ 是 → Workflow（例如：代码审查必须通过 7 分门槛）

└─ 否 → 继续 Q3

Q3: 这个任务是否可能运行超过 5 分钟，需要中断恢复？

├─ 是 → Workflow（例如：全仓库迁移、批量重构）

└─ 否 → 继续 Q4

Q4: 这个任务是否是"教 Agent 如何做"（知识注入）而非"指挥 Agent 做什么"（编排）？

├─ 知识注入 → Skill（例如：后端架构模式、编码规范）

└─ 编排任务 → Workflow（例如：多步骤开发流程）

快速判断表

如果你的任务是...用理由
"帮我写一个 REST API 端点"Skill单 Agent 任务，Skill 提供 API 设计模式知识
"按照公司规范审查这个 PR"Skill审查规范是知识，单个 Agent 执行审查
"对整个仓库做安全审计"Workflow需要并行分析多个模块，交叉验证结果
"学习这个新代码库的架构"Skill探索性任务，Agent 自由探索，Skill 提供分析方法
"从 JS 迁移到 TS，共 200 个文件"Workflow批量任务，需要并行处理、检查点、恢复机制
"部署前做最终检查"Workflow需要强制性的检查清单，不合格必须阻止
"修复这个 Bug"Skill单 Agent 任务，Skill 提供 Bug 修复流程指导
"同时优化后端性能和前端加载速度"Workflow两个独立任务，可以并行执行
组合使用
Skill 和 Workflow 不是互斥的。最佳实践是：用 Skill 提供领域知识，用 Workflow 编排执行流程。例如，一个"全栈代码审查" Workflow 可以：
  • Phase 1: 后端审查 Agent 使用 backend-patterns Skill
  • Phase 2: 前端审查 Agent 使用 frontend-patterns Skill
  • Phase 3: 安全审查 Agent 使用 security-review Skill
  • Phase 4: 汇总审查结果，质量评分

12.6 实战案例：Skill 与 Workflow 的典型场景

案例 1：Bug 修复流程 (Skill)

本项目使用 bug-feature-dev Skill 来标准化 Bug 修复和功能开发流程。这个 Skill 定义了 6 个步骤：需求分析 → 更新 ROADMAP → 判断 API 变更 → 前后端并行实现 → 验证 → 标记完成。每个步骤都是对当前 Agent 的指导，由 Agent 自行判断如何执行。

name: bug-feature-dev
description: |
  Multi-Agent Platform 项目的 Bug 修复与功能开发工作流。
  当用户报告 bug、提出功能需求、
  或说"修复xxx"、"新增xxx功能"、"优化xxx"时触发此 Skill。

# Bug / Feature 工作流

## 步骤 1: 需求分析
1. 复述需求 — 用自己的话确认理解是否正确
2. 判断影响范围 — 涉及哪些模块？
3. 列出潜在风险 — 可能破坏现有功能吗？

## 步骤 2: 更新 ROADMAP
...

## 步骤 3: 判断 API 变更
...

## 步骤 4: 前后端并行实现
将任务拆分为两个独立的子 Agent...

## 步骤 5: 验证
go build ./... && vue-tsc --noEmit && vite build

## 步骤 6: 更新 ROADMAP 标记完成
为什么用 Skill 而不是 Workflow？
这个流程虽然有多步骤，但本质上是单 Agent 的顺序执行。Agent 按照 Skill 中的指导逐步完成，不需要多个 Agent 并行。步骤 4 的"前后端并行实现"是指导 Agent 使用子 Agent 工具，而非 Workflow 层面的编排。

案例 2：多模型协作开发 (Workflow)

multi-workflow 是一个典型的 Workflow 案例，它编排了 6 个 Phase 的协作开发流程，每个 Phase 中可能并行调用多个外部模型。

// multi-workflow 的核心结构

// Phase 1: 研究分析
// - 提示增强（可选 MCP）
// - 上下文检索
// - 需求完整性评分（≥7 分才继续）

// Phase 2: 方案构思（并行）
// - Codex: 技术可行性 + 方案 + 风险
// - Gemini: UI 可行性 + 方案 + UX 评估
// → 综合两个模型的输出，提供至少 2 个方案选项

// Phase 3: 详细规划（并行）
// - Codex: 后端架构
// - Gemini: 前端架构
// → Claude 综合：采用 Codex 后端 + Gemini 前端

// Phase 4: 实现
// - 严格遵循批准的计划
// - 遵循项目代码规范

// Phase 5: 优化（并行）
// - Codex: 安全、性能、错误处理
// - Gemini: 可访问性、设计一致性

// Phase 6: 质量审查
// - 检查完成度
// - 运行测试
// - 评分 < 7 或用户不批准 → 强制停止
为什么用 Workflow 而不是 Skill？
这个流程需要：(1) 并行调用多个外部模型（Codex + Gemini 同时工作）；(2) 强制性的质量门控（评分 < 7 必须停止）；(3) 阶段间的数据传递（Phase 2 的输出影响 Phase 3 的输入）；(4) 用户确认点（每个 Phase 完成后等待用户确认）。这些都不是单个 Skill 能实现的。

案例 3：Deep Research (Skill + 内部并行)

deep-research Skill 展示了一个有趣的混合模式：它本身是一个 Skill（Markdown 文件），但内部指导 Agent 使用子 Agent 工具做并行研究。

## Parallel Research with Subagents

For broad topics, use Claude Code's Task tool to parallelize:

Launch 3 research agents in parallel:
1. Agent 1: Research sub-questions 1-2
2. Agent 2: Research sub-questions 3-4
3. Agent 3: Research sub-question 5 + cross-cutting themes

Each agent searches, reads sources, and returns findings.
The main session synthesizes into the final report.
这是 Skill 还是 Workflow？
这是 Skill 指导下的 Agent 内部并行。Skill 告诉 Agent "你可以用子 Agent 并行研究"，但实际的并行执行是由 Agent 在对话中自行发起的，不是 Workflow 引擎层面的编排。这种模式适合需要灵活性的探索性任务——不需要严格的质量门控，但可以从并行中受益。

# Go Backend Development Standards

## When to Activate
- Writing or modifying Go code in internal/ or pkg/
- Adding new API endpoints
- Working with SQLite database
- Implementing new tools or agents
- Debugging concurrency issues

## Project Structure Conventions
- cmd/server/ — Entry point, HTTP handlers, route registration
- internal/ — Business logic (agent, llm, runtime, tool, ws)
- pkg/ — Shared packages (event, db)
- New features go in internal/; shared utilities in pkg/

## Error Handling
```go
// Always wrap errors with context
if err != nil {
    return fmt.Errorf("failed to execute tool %s: %w", toolName, err)
}
```

## Concurrency
- Use sync.RWMutex for read-heavy shared state
- Use context.Context for cancellation propagation
- Goroutine leaks are bugs — always ensure cleanup

## Database
- Schema changes via pkg/db/migrate.go — append, never modify
- Use prepared statements for all queries
- Transaction for multi-table writes

8.2 完整 Workflow 示例：多 Agent 代码审查

// 文件名: multi-agent-code-review.workflow.js
// 用途: 对整个仓库进行多视角、多 Agent 的并行代码审查

export const meta = {
    name: "multi-agent-code-review",
    description: "Parallel multi-perspective code review with adversarial verification",
    budget: {
        total: 150000,      // 15万 token 预算
        perAgent: 40000,     // 每个 Agent 4万 token
        maxTime: 900,        // 15分钟超时
    },
    resume: true,
    cache: "phase",
};

// ============================================================
// Phase 1: 代码库扫描
// ============================================================
phase("Scan", async (ctx) => {
    log.info("开始扫描代码库结构");

    const scanResult = await agent({
        name: "codebase-scanner",
        prompt: `扫描项目代码库，识别所有模块和文件：
        - 列出所有 Go 包及其职责
        - 列出所有前端组件及其职责
        - 识别跨模块的依赖关系
        - 标记可能的问题区域（大文件、循环依赖、缺少测试）`,
        model: "haiku",       // 扫描用轻量模型
        tools: ["glob", "grep", "read"],
        maxSteps: 20,
    });

    ctx.data.modules = parseModuleList(scanResult.output);
    ctx.data.totalFiles = ctx.data.modules.length;

    log.info("扫描完成", {
        modules: ctx.data.modules.length,
        tokens: scanResult.tokens,
    });
});

// ============================================================
// Phase 2: 并行多视角审查
// ============================================================
phase("Review", async (ctx) => {
    log.info("启动并行审查");

    // 并行启动 4 个审查 Agent
    const [securityReview, performanceReview, styleReview, bugReview] =
        await parallel([
            // Agent 1: 安全审查
            agent({
                name: "security-reviewer",
                prompt: `审查以下模块的安全问题：
                ${JSON.stringify(ctx.data.modules)}
                关注: SQL注入、路径遍历、未校验输入、密钥泄露、不安全的依赖`,
                model: "sonnet",
                tools: ["read", "grep"],
                maxSteps: 25,
            }),

            // Agent 2: 性能审查
            agent({
                name: "performance-reviewer",
                prompt: `审查以下模块的性能问题：
                ${JSON.stringify(ctx.data.modules)}
                关注: 不必要的内存分配、goroutine泄漏、阻塞IO、N+1查询`,
                model: "sonnet",
                tools: ["read", "grep"],
                maxSteps: 25,
            }),

            // Agent 3: 代码风格审查
            agent({
                name: "style-reviewer",
                prompt: `审查以下模块的代码风格：
                ${JSON.stringify(ctx.data.modules)}
                关注: 命名规范、注释完整性、函数长度、文件组织、错误处理一致性`,
                model: "haiku",     // 风格审查用轻量模型
                tools: ["read", "grep"],
                maxSteps: 20,
            }),

            // Agent 4: Bug 模式审查
            agent({
                name: "bug-pattern-reviewer",
                prompt: `审查以下模块的常见 Bug 模式：
                ${JSON.stringify(ctx.data.modules)}
                关注: 空指针、竞态条件、资源泄漏、边界条件、错误吞没`,
                model: "sonnet",
                tools: ["read", "grep"],
                maxSteps: 25,
            }),
        ]);

    // 保存审查结果
    ctx.data.reviews = {
        security: securityReview,
        performance: performanceReview,
        style: styleReview,
        bugs: bugReview,
    };

    const totalIssues = countIssues(ctx.data.reviews);
    log.info("并行审查完成", { totalIssues });
});

// ============================================================
// Phase 3: 对抗性验证
// ============================================================
phase("Verify", async (ctx) => {
    log.info("开始对抗性验证");

    // 第二个 Agent 独立验证安全审查的结果
    const verification = await agent({
        name: "security-verifier",
        prompt: `独立验证以下安全审查发现的每个问题：
        ${ctx.data.reviews.security.output}

        对于每个发现:
        - 确认: 问题确实存在 → 标记 CONFIRMED
        - 误报: 问题不存在 → 标记 FALSE_POSITIVE，说明原因
        - 不确定: 无法确认 → 标记 UNCERTAIN，说明需要什么额外信息`,
        model: "sonnet",
        tools: ["read", "grep"],
        maxSteps: 15,
    });

    ctx.data.verifiedFindings = verification.output;

    const { confirmed, falsePositives } = parseVerification(verification.output);
    log.info("验证完成", { confirmed, falsePositives });
});

// ============================================================
// Phase 4: 生成报告
// ============================================================
phase("Report", async (ctx) => {
    const report = await agent({
        name: "report-generator",
        prompt: `基于以下审查结果生成综合报告：

        安全审查: ${ctx.data.reviews.security.output}
        性能审查: ${ctx.data.reviews.performance.output}
        风格审查: ${ctx.data.reviews.style.output}
        Bug审查: ${ctx.data.reviews.bugs.output}
        验证结果: ${ctx.data.verifiedFindings}

        报告格式:
        # 代码审查报告
        ## 概览
        ## 安全问题 (按严重程度排序)
        ## 性能问题
        ## 代码风格问题
        ## Bug 模式
        ## 建议的修复优先级`,
        model: "sonnet",
        maxSteps: 10,
    });

    // 质量评分
    const score = calculateQualityScore(ctx.data.reviews);

    if (score < 7) {
        log.error("质量评分不达标", { score });
        throw new Error(`代码质量评分 ${score}/10，低于 7 分门槛，审查终止`);
    }

    ctx.data.finalReport = report.output;
    ctx.data.qualityScore = score;

    log.info("报告生成完成", { score, tokens: report.tokens });
});
这个 Workflow 展示了什么
  • Phase 分层：扫描 → 审查 → 验证 → 报告，每个阶段有明确的职责
  • 并行执行：Phase 2 中 4 个审查 Agent 同时运行，节省 75% 时间
  • 对抗性验证：Phase 3 中第二个 Agent 独立验证安全审查的结果
  • 质量门控：Phase 4 中评分 < 7 自动终止
  • 预算控制：meta.budget 定义了总预算和每 Agent 预算
  • 数据传递：ctx.data 在 Phase 间持久化数据
  • 结构化日志：每个关键步骤都有 log 输出
