多模型分层设计 设计文档

多 Agent 场景下的模型分层策略、Router + Worker Pool 架构设计、成本优化方案
适用于 Phase 4-6 的多 Agent 并发与模型管理

📋 文档目的
本文档定义多 Agent 系统中模型分层策略和路由调度架构。 核心目标：在保证 Agent 能力的前提下，通过分层模型选择将 API 调用成本控制在合理范围。

1. 模型分层概览

多 Agent 系统中，不同 Agent 的任务复杂度差异巨大——从简单的意图分类到复杂的多步推理。 单一模型无法同时满足成本控制和能力上限两个维度。 因此引入 6 层模型分层：

🔴 顶层 — 复杂推理/编排决策 🟠 高层 — 主力 Agent 执行 🟡 中层 — 并行 Agent / 代码生成 🟢 轻量 — 快速分类/路由 🟣 高效 — 大量吞吐/低成本 ⚫ 免费 — 测试/冷备/本地开发
层 代表模型 输入价格 输出价格 定位 典型场景
顶层 Claude Opus 4.5/4.6 $5/M $25/M 复杂推理 / 编排决策 多步任务规划、自反思修正、复杂代码架构设计
高层 Sonnet 4.5/4.6 $3/M $15/M 主力 Agent 执行 通用任务执行、Tool Calling、代码生成
中层 GPT-5.4 · Gemini 3.1 Pro · GPT-5.3-Codex $1.75-2.5/M $12-15/M 并行 Agent / 代码生成 批量代码生成、长上下文 RAG、多 Agent 并行执行
轻量 Claude Haiku 4.5 $1/M $5/M 快速分类 / 路由 意图识别、请求分类、简单问答、格式转换
高效 DeepSeek V4-Flash/Pro $0.14-1.71/M $0.29-3.43/M 大量吞吐 / 低成本 批量分析、结果校验、数据清洗、简单代码生成
免费 DeepSeek-V4-Flash-Local · Qwen-397B · GLM · Kimi Free Free 测试 / 冷备 / 本地开发 本地调试、单元测试、CI/CD 流水线、灾难降级
💡 当前项目状态
当前（Phase 1-2）使用内部 API 的 deepseek-v4-flash（高效层）， 成本极低。后续接入外部模型时，按此分层逐步引入。

2.1 各组件职责

组件模型职责延迟要求
Router Haiku / DeepSeek-Flash 意图识别、复杂度评估、请求分类、模型选择 < 500ms
Planner Opus 复杂任务分解、子任务编排、依赖关系管理 可容忍 5-10s
Worker Sonnet / Codex / Gemini 执行具体子任务：代码生成、工具调用、分析推理 1-5s
Validator DeepSeek-Flash 结果校验、安全检查、输出格式化 < 1s

3. 成本优化分析

3.1 优化公式

策略机制预估节省
分层路由 简单请求走 Haiku/Flash，复杂请求才走 Sonnet/Opus 省 70-80%
批量 Flash 大量分析任务全走 DeepSeek-Flash 省 95%+
本地开发 调试阶段用免费本地模型 省 100%
廉价互检 Validator 用 Flash 而非 Sonnet 省 90%+

3.2 成本计算示例

假设一天 1000 次请求，其中 70% 简单、30% 复杂：

// 方案 A: 全用 Sonnet（无分层） 1000 × ~2000 tokens × $3/M input + 1000 × ~500 tokens × $15/M output = $6.00 + $7.50 = $13.50/天 // 方案 B: 分层路由 Router: 1000 × ~100 tokens × $1/M = $0.10 简单: 700 × ~2000 tokens × $0.14/M + 700 × ~500 × $0.29/M = $0.20 + $0.10 = $0.30 复杂: 300 × ~2000 tokens × $3/M + 300 × ~500 × $15/M = $1.80 + $2.25 = $4.05 Validator: 300 × ~500 tokens × $0.14/M = $0.02 总计: $0.10 + $0.30 + $4.05 + $0.02 = $4.47/天 // 节省: ($13.50 - $4.47) / $13.50 = 67%

4. 缺失维度补充

⚠️ 基础分层设计缺少以下关键维度
仅有模型价格和定位是不够的。实际工程中还需要考虑能力矩阵、降级策略、上下文窗口、延迟约束。

4.1 能力矩阵

不是所有模型都支持所有能力。Router 必须根据 任务所需能力 筛选可用模型：

Claude Opus 4.5 ✅✅✅✅✅
Claude Sonnet 4.5 ✅✅✅✅✅
Claude Haiku 4.5 ✅✅✅⚠️ 有限✅
DeepSeek V4-Flash ✅✅❌⚠️ 有限✅
DeepSeek R1 ⚠️ 部分✅❌✅ 强✅
GPT-5.3-Codex ✅✅❌⚠️ 有限✅

4.2 降级策略

每个模型层必须有 fallback，API 不可用是常态：

主模型降级模型降级条件

DeepSeek Flash→ 本地免费模型API 超时 / 限流
所有云端模型→ 本地 Qwen/GLM网络中断 / 全局故障

Router 必须根据输入长度排除不合适的模型：

5. ModelProfile 抽象设计

ModelProfile 是模型分层的核心数据结构，描述一个模型的完整画像——能力、成本、限制、降级路径。

// ModelProfile 描述一个模型的完整画像
// 用于 Router 做模型选择决策和 Worker Pool 做能力匹配
type ModelProfile struct {
    // 基础标识
    Name     string    // "deepseek-v4-flash"
    Provider string    // "deepseek" / "openai" / "anthropic"
    Tier     ModelTier  // 分层标识

    // 能力矩阵
    Capabilities []ModelCapability // tools / vision / reasoning / streaming / json_mode

    // 成本 (per 1M tokens, USD)
    InputPrice  float64
    OutputPrice float64

    // 上下文限制
    MaxContextWindow int  // 最大上下文 token 数
    MaxOutputTokens  int  // 最大输出 token 数

    // 速率限制
    RateLimitRPM int  // 每分钟最大请求数

    // 降级路径
    FallbackModel string  // 当前模型不可用时的替代模型名

    // 延迟特征
    AvgLatencyMs int  // 平均响应延迟
}

type ModelTier int
const (
    TierFree       ModelTier = iota  // ⚫ 免费/本地
    TierEfficient                        // 🟣 低成本高吞吐
    TierLightweight                      // 🟢 快速分类路由
    TierStandard                         // 🟠 主力执行
    TierPremium                          // 🔴 复杂推理
)

type ModelCapability string
const (
    CapToolCalling ModelCapability = "tool_calling"
    CapStreaming   ModelCapability = "streaming"
    CapVision      ModelCapability = "vision"
    CapReasoning   ModelCapability = "reasoning"  // 深度推理/思维链
    CapJSONMode    ModelCapability = "json_mode"
)

5.1 ModelRegistry — 全局模型注册表

// ModelRegistry 管理所有可用模型的 Profile
// 从配置文件或数据库加载，支持运行时动态更新
type ModelRegistry struct {
    profiles  map[string]*ModelProfile  // name → profile
    byTier    map[ModelTier][]string    // tier → model names
    mu        sync.RWMutex
}

// 常用方法
func (r *ModelRegistry) GetByTier(tier ModelTier) []*ModelProfile
func (r *ModelRegistry) FilterByCapability(cap ModelCapability) []*ModelProfile
func (r *ModelRegistry) FilterByContextLen(minTokens int) []*ModelProfile
func (r *ModelRegistry) GetFallback(name string) *ModelProfile

6. Router 路由设计

Router 是意图识别 + 模型选择的核心组件。路由决策本身也由 LLM 完成，但使用最廉价的模型。

6.1 路由决策流程

// 路由决策 pipeline RouteRequest │ ▼ // Step 1: 规则过滤（零成本，零延迟） ContextLen > MaxContextWindow? → 排除该模型 需要 Tool Calling 但模型不支持? → 排除该模型 延迟要求高但模型慢? → 排除该模型 │ ▼ // Step 2: 意图分类（轻量模型，~100 tokens，成本 < $0.001） Classifier: "Classify this request into: simple_chat | code_gen | complex_reasoning | multi_step" │ ▼ // Step 3: 按意图 + 预算 + 延迟 → 选择最佳模型 simple_chat → TierEfficient (DeepSeek-Flash) code_generation → TierStandard (Sonnet) complex_reasoning → TierPremium (Opus) multi_step → TierStandard (Sonnet) + TierPremium (Planner) │ ▼ RouteDecision { Primary, Fallback, Reason }

6.2 Router 接口

// Router 根据请求特征选择最优模型
type Router struct {
    registry   *ModelRegistry
    classifier *Client  // 用于意图分类的轻量模型 (Haiku/Flash)
}

// RouteRequest 是路由器的输入
type RouteRequest struct {
    UserInput   string         // 用户原始输入
    ContextLen  int            // 输入 token 数
    RequiredCaps []ModelCapability // 必需的能力
    Budget      float64        // 可选：成本上限 (USD)
    LatencyReq  time.Duration  // 可选：延迟要求
}

// RouteDecision 是路由器的输出
type RouteDecision struct {
    Primary   *ModelProfile  // 首选模型
    Fallback  *ModelProfile  // 降级模型
    Reason    string         // 路由原因（白盒展示）
    Tier      ModelTier      // 选中的分层
}

7. Worker Pool 设计

Worker Pool 管理多 Agent 并发执行，核心职责：模型分配、并发控制、成本追踪、降级处理。

7.1 接口设计

// WorkerPool 管理多 Agent 的并发执行
type WorkerPool struct {
    router    *Router
    registry  *ModelRegistry
    agents    map[string]*AgentRuntime  // agentID → runtime
    semaphore chan struct{}             // 并发槽位
    tracker   *CostTracker              // 成本追踪
    mu        sync.RWMutex
}

// Dispatch 分配任务到合适的 Agent
func (p *WorkerPool) Dispatch(task Task) error {
    // 1. Router 选择模型
    decision := p.router.Select(&RouteRequest{
        UserInput:   task.UserInput,
        ContextLen:  estimateTokens(task.Messages),
        RequiredCaps: requiredCaps(task),
    })

    // 2. 发送路由决策事件（白盒展示）
    p.emitEvent(EventRouterDecision{
        TaskID:   task.ID,
        Primary:  decision.Primary.Name,
        Fallback: decision.Fallback.Name,
        Reason:   decision.Reason,
    })

    // 3. 获取/创建 Agent 实例
    agent := p.getOrCreateAgent(task.ID, decision.Primary)

    // 4. 执行 + 监控
    ctx, cancel := context.WithTimeout(context.Background(), task.Timeout)
    defer cancel()

    result, err := agent.Run(ctx, task)
    if err != nil {
        // 5. 降级重试
        if decision.Fallback != nil && isRetryable(err) {
            log.Warn("primary model failed, falling back",
                "primary", decision.Primary.Name,
                "fallback", decision.Fallback.Name,
                "error", err,
            )
            agent = p.getOrCreateAgent(task.ID, decision.Fallback)
            result, err = agent.Run(ctx, task)
        }
    }

    // 6. 记录成本
    p.tracker.Record(CostRecord{
        TaskID:   task.ID,
        Model:    decision.Primary.Name,
        Tier:     decision.Tier,
        Tokens:   result.Usage,
        Cost:     p.calculateCost(decision.Primary, result.Usage),
    })

    return err
}

8.2 演进路径

阶段改动范围破坏性说明
Phase 1-2（当前） 单一 Client，硬编码端点 — 保持现状，不做架构变更
Phase 3-4 新增 model_profile.go + 配置加载 无 纯新增，不影响现有流程。Agent 创建时可选指定模型

9. 实施优先级

🎯 核心原则：架构跟着需求走，不提前抽象
当前阶段（Phase 2-3）的核心任务是前端可视化和预设 Cases，保持单一模型。 多模型分层是 Phase 4-6 的需求，届时对 LLM 领域的理解更深，接口设计会更成熟。
阶段交付物理由
Phase 3-4 ModelProfile 类型 + ModelRegistry
从配置/DB 加载模型列表
Agent 创建时可选指定模型 		为 Router 做数据准备，纯新增，不影响现有功能。
多 Agent 并发自然需要不同模型。
Phase 5 Provider 接口抽取
Router 意图分类 + 模型选择
openai_provider.go 基线实现 		此时 Provider 抽象足够成熟，
Router 依赖 ModelRegistry 和 Provider 两层。
Phase 6 WorkerPool 并发调度
CostTracker 成本追踪
降级策略 + 自动 fallback
AnthropicProvider 实现 		生产级特性，依赖前面所有层。
成本追踪是白盒 Agent 的最终闭环。
